# Verification

This page is the current human-readable correctness snapshot for `ShinBucketDeployment`. Performance evidence and AWS CDK `BucketDeployment` comparisons remain separate in [benchmark](./benchmark.md). Verification is replaced rather than appended; runbooks and sanitization rules live in `.agents/skills/shin-verification/SKILL.md`.

> [!IMPORTANT]
> The released `shin-bucket-deployment@0.7.1` package was installed from the npm registry into a clean scratch CDK app and deployed in `eu-central-1` with its packaged x86_64 provider. The stack reached `CREATE_COMPLETE`, the deployed Lambda reported `x86_64`, and the destination object matched the exact expected body. The stack and bucket were destroyed and independently confirmed absent. This smoke verifies the released artifact, not the unreleased changes currently on `main`.
>
> The current construct-API candidate has complete deterministic local verification for grouped CloudFront invalidation, explicit phase-local destination lifecycle actions, CDK-style provider-scope, failure-diagnostics, and retry-jitter enums, local-provider build settings, renamed advanced tuning, and demand-driven object-key output. The lifecycle contract uses `deleteStaleObjects`, `deletePreviousObjects`, `previousBucket`, `invalidatePreviousDistribution`, and `deleteCurrentObjects`; the shorter ambiguous names fail with migration guidance. The generated custom-resource protocol, provider environment, IAM grants, and Lambda topology remain covered by exact synthesis assertions; no AWS run was performed because the changed boundaries are decided before deployment while the Rust data path is unchanged.
>
> [PR #35](https://github.com/garysassano/shin-bucket-deployment/pull/35) has complete deterministic local verification for destination-prefix ownership validation. Concrete prefixes accept the exact 102-character boundary and reject 103 characters with a prefix-specific error; unresolved tokens fail before provider resources are created; and `"/"` synthesizes the same canonical root ownership tag as an omitted prefix. No AWS run was performed because every changed boundary is decided during construct creation or synthesis, before deployment can begin. The latest current-code targeted AWS evidence remains the CopyObject create and tamper-repair run. The broader integration evidence remains recorded in the earlier component snapshots, and the expanded 33-phase matrix has not yet been rerun as one combined AWS pass. Raw AWS output and identifiers remain outside git.

## Known Limitations

- Marker uploads require one bounded planning pass for exact length and validation, followed by a second bounded source pass when an upload is required; unchanged SSE-S3 objects stop after planning. The separately maintained performance decision run measures and accepts this tradeoff.
- The released AWS S3 dependency chain currently requires newly yanked `spin@0.10.0` through `crc-fast`. The full lockfile audit reports that yanked warning and no vulnerability; the dependency-policy command excludes only that exact transitive version until `crc-fast` publishes a compatible non-yanked dependency.
- The transfer-scheduler decision run contains five before, five current, and four upstream repetitions; the maintainer stopped the final upstream repetition to cap time and accepted the completed performance evidence for PR review. The 112 sanitized phase rows are retained in `benchmarks/results.jsonl`.
- The released npm `0.7.1` x86_64 provider passed a separate registry-installed AWS deployment. That release smoke does not validate the unreleased provider changes now on `main`; current-code targeted AWS runs used arm64, while both current archives passed structural package verification.
- The invocation-global source cap governs both conservative central-directory planning estimates and retained source blocks in non-overlapping phases. Runtime, SDK, transfer-body, manifest, and marker allocations use the other half of Lambda memory and remain bounded by concurrency and object-size rules.
- Destination metadata is O(manifest keys), and stale cleanup is O(one S3 page); the second destination scan runs only when the initial listing identifies an exact included non-manifest candidate. The deployment manifest itself remains proportional to planned object count.
- Lost-response convergence is covered with deterministic S3 wire replay rather than an injected AWS network failure. The live direct-copy run covered ordinary Create and guarded repair after an independent overwrite, but did not inject a lost response. The real KMS/DSSE run confirms that the stored checksum needed by upload reconciliation exists and matches.
- KMS/DSSE destination ETags are not treated as plaintext MD5. Existing encrypted objects can transfer again instead of using the SSE-S3 catalog/ETag shortcut; avoiding a checksum-mode `HeadObject` per destination object keeps the normal deployment request count bounded.
- Deployments remain non-transactional across objects. A marker upload withholds its final body frame until the second pass validates source integrity and planned output identity, but earlier independent object writes may already have completed before a later object fails.
- Imported buckets, SSE-C, and encryption configurations whose strategy cannot be proven at synthesis remain intentionally unsupported.
- Raw AWS evidence remains intentionally excluded from git. Performance results and their separate caveats are maintained in [benchmark](./benchmark.md).
