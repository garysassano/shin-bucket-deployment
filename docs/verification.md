# Verification

This page is the current human-readable correctness snapshot for `ShinBucketDeployment`. Performance evidence and AWS CDK `BucketDeployment` comparisons remain separate in [benchmark](./benchmark.md). Verification is replaced rather than appended; runbooks and sanitization rules live in `.agents/skills/shin-verification/SKILL.md`.

> [!IMPORTANT]
> The current release candidate keeps one strict custom-resource protocol for Create, Update, and Delete: the modeled resource type and complete current request schema are required for every operation. The preceding implementation candidate passed a targeted AWS `object-deletion` run whose ordered Create, destination-change Update, and custom-resource Delete phases all used that current full request shape and passed SDK-backed exact-state assertions. The terminal stack and captured bucket were then destroyed and independently confirmed absent. Raw AWS output and identifiers remain outside git.
>
> [PR #64](https://github.com/garysassano/shin-bucket-deployment/pull/64) closes the H3/H4 workflow assurance gaps without changing the provider data path. The checked-in benchmark workflow now uses valid sequential GitHub Actions steps and raw `actionlint`; the syntax-rewriting lint shim is gone. For correctness, every one of the 36 cataloged verification phases has an SDK-backed expected-state assertion, deployment outputs persist outside CDK assembly directories, assertion failures fail the runner, and cleanup verifies stack and captured-resource absence. A partial AWS smoke at `f3b5050` completed seven independent groups with no deployment assertion failure, but the hosted run was deliberately canceled after its shared cleanup probe required account-wide bucket-list permission. Exact residual scenario stacks were subsequently destroyed and no scenario stack or bucket created during the run window remained. This is partial smoke evidence, not a complete combined AWS pass. The hosted full-matrix workflow has been removed; AWS verification is now deliberately maintainer-run through targeted named groups, selected independent groups, or the rare full suite. No benchmark refresh was required because neither the provider bytes nor its performance-sensitive data path changed.
>
> [PR #62](https://github.com/garysassano/shin-bucket-deployment/pull/62) has complete protocol and AWS regression evidence for destination changes at implementation commit `f41dde3`. Create retains the destination-derived physical ID needed for handler-generation safety, while Update returns the incoming physical ID so bucket and prefix changes remain ordinary updates governed only by `destinationLifecycle.onChange`. The ordered AWS chain covered child-to-parent, parent-to-child, sibling-prefix, and cross-bucket moves with `onDelete.deleteCurrentObjects=true` and both values of `onChange.deletePreviousObjects`. Both phases passed exact object-body and deletion assertions, the original handler-memory replacement remained covered, and the stack was destroyed and confirmed absent. Raw AWS output and identifiers remain outside git.
>
> The released `shin-bucket-deployment@0.7.1` package was installed from the npm registry into a clean scratch CDK app and deployed in `eu-central-1` with its packaged x86_64 provider. The stack reached `CREATE_COMPLETE`, the deployed Lambda reported `x86_64`, and the destination object matched the exact expected body. The stack and bucket were destroyed and independently confirmed absent. This smoke verifies the released artifact, not the unreleased changes currently on `main`.
>
> The `0.10.0` API candidate has complete deterministic local verification for all seven root components, strict unknown-property rejection at every nested level, complete migration guidance for the former flat props, exported defaults, and canonical provider identity. Provider sharing uses `providerLambda.sharing: ProviderSharing`; request-level transfer concurrency uses `transfer.maxConcurrency`; and lifecycle actions remain phase-oriented under `destinationLifecycle`. Before the release version bump, every synthesized assembly JSON artifact for all 33 verification phases matched `origin/main`. Both benchmark apps differed only by the renamed `BenchmarkTransferMaxConcurrency` output node and its derived template asset hash references. The `0.10.0` package version intentionally selects a new shared-handler identity; the custom-resource protocol, provider environment, packaged provider bytes, and Rust data path are unchanged. No AWS run was performed for these naming and release-metadata changes.
>
> [PR #35](https://github.com/garysassano/shin-bucket-deployment/pull/35) has complete deterministic local verification for destination-prefix ownership validation. Concrete prefixes accept the exact 102-character boundary and reject 103 characters with a prefix-specific error; unresolved tokens fail before provider resources are created; and `"/"` synthesizes the same canonical root ownership tag as an omitted prefix. No AWS run was performed because every changed boundary is decided during construct creation or synthesis, before deployment can begin. The broader integration evidence remains recorded in the earlier component snapshots, and the current 36-phase suite has not been rerun as one complete combined AWS pass. Raw AWS output and identifiers remain outside git.

## Known Limitations

- PR #64 does not have a complete combined AWS assertion pass. Seven independent groups passed their deployment assertions at `f3b5050`; the hosted run was canceled because its shared cleanup probe required account-wide bucket-list authorization. Cleanup was completed separately. The hosted matrix was removed in favor of deliberate maintainer-run groups, and no rerun was launched merely to replace partial evidence with a green badge.
- Marker uploads require one bounded planning pass for exact length and validation, followed by a second bounded source pass when an upload is required; unchanged SSE-S3 objects stop after planning. The separately maintained performance decision run measures and accepts this tradeoff.
- The released AWS S3 dependency chain currently requires newly yanked `spin@0.10.0` through `crc-fast`. The full lockfile audit reports that yanked warning and no vulnerability; the dependency-policy command excludes only that exact transitive version until `crc-fast` publishes a compatible non-yanked dependency.
- The transfer-scheduler decision run contains five before, five current, and four upstream repetitions; the maintainer stopped the final upstream repetition to cap time and accepted the completed performance evidence for PR review. The 112 sanitized phase rows are retained in `benchmarks/results.jsonl`.
- The released npm `0.7.1` x86_64 provider passed a separate registry-installed AWS deployment. That release smoke does not validate the unreleased provider changes now on `main`; current-code targeted AWS runs used arm64, while both current archives passed structural package verification.
- The invocation-global source cap governs both conservative central-directory planning estimates and retained source blocks in non-overlapping phases. Runtime, SDK, transfer-body, manifest, and marker allocations use the other half of Lambda memory and remain bounded by concurrency and object-size rules.
- Destination metadata is O(manifest keys), and stale cleanup is O(one S3 page); the second destination scan runs only when the initial listing identifies an exact included non-manifest candidate. The deployment manifest itself remains proportional to planned object count.
- Lost-response convergence is covered with deterministic S3 wire replay rather than an injected AWS network failure. The live direct-copy run covered ordinary Create and guarded repair after an independent overwrite, but did not inject a lost response. The real KMS/DSSE run confirms that the stored checksum needed by upload reconciliation exists and matches.
- KMS/DSSE destination ETags are not treated as plaintext MD5. Existing encrypted objects can transfer again instead of using the SSE-S3 catalog/ETag shortcut; avoiding a checksum-mode `HeadObject` per destination object keeps the normal deployment request count bounded.
- Deployments remain non-transactional across objects. A marker upload withholds its final body frame until the second pass validates source integrity and planned output identity, but earlier independent object writes may already have completed before a later object fails.
- Imported buckets, SSE-C, and encryption configurations whose strategy cannot be proven at synthesis remain intentionally unsupported.
- Raw AWS evidence remains intentionally excluded from git. Performance results and their separate caveats are maintained in [benchmark](./benchmark.md).
